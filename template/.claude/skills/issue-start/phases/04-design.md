# Phase 4: 設計（code-architect）

**目的**: 複数の設計案を比較し、最適なアプローチをユーザーと合意してから実装に入る。事前の設計合意が手戻りを防ぐ。

## 実行手順

1. Phase 3の探索結果を踏まえ、**`code-architect` サブエージェントを2〜3個並列で起動**する
   - `Agent` ツールで `subagent_type: "code-architect"` を指定する
   - エージェント定義は `.claude/agents/code-architect.md` に集約済み
2. 各エージェントには異なるアプローチで設計させる：
   - **最小変更**: 「既存コードへの変更を最小限にし、最大限再利用するアプローチで設計して」
   - **クリーン設計**: 「保守性・テスタビリティを重視した理想的なアーキテクチャで設計して」
   - **実用的バランス**: 「実装速度とコード品質のバランスを取るアプローチで設計して」
3. 全アプローチを比較し、**推奨案を理由付きでユーザーに提示**する
4. **ユーザーの承認を得てから**次のPhaseに進む

## 提示フォーマット

```
## 設計案の比較

### 案1: 最小変更
- 変更ファイル: 2個
- メリット: 低リスク、早い
- デメリット: 既存コードとの結合度が高い

### 案2: クリーン設計
- 変更ファイル: 5個（新規2個）
- メリット: テストしやすい、拡張性が高い
- デメリット: 工数が多い

### 案3: 実用的バランス（推奨）
- 変更ファイル: 3個（新規1個）
- メリット: 適度な分離、現実的な工数
- デメリット: 一部結合が残る

→ 案3を推奨します。理由: ...

どのアプローチで進めますか？
```

## 必須チェック項目（同系統要素の網羅性）

「同じ扱いをすべき類似要素のグループ」（例: 同系統のデータ定義群・同じインターフェースを実装するクラス群・同種のイベントハンドラ群）に対して新しい仕様・ルールを適用する設計の場合、**グループ内の全要素が新仕様の対象になっているか** を必ず設計時に検証する。

確認手順:

1. 対象グループの全要素を `Grep` で列挙する（例: `Grep -n "implements SomeInterface"` / `Grep -n "id:.*<系統キーワード>"`）
2. 各要素が新仕様の経路を通るか、`code-architect` の設計案レベルで確認する
3. 一部しか対象にならない設計案の場合、**設計の段階で** グループ横断テストの要否をユーザーに合意しておく（Phase 5 に伝播）

> **背景**: グループの代表 1 要素だけ確認して設計を進めると、残りの要素で新仕様が反応しない手戻りが起きやすい。「1つ動いた＝全部動く」とは限らない。

## スキップ条件

Phase 3と同じ条件に加え：
- 変更が1〜2ファイルで収まる明確な修正の場合
- Issueに具体的な実装方針が既に記載されている場合
