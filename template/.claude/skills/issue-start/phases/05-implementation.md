# Phase 5: 実装

1. `CLAUDE.md` および `.claude/rules/coding-standards.md` のコーディング規約に従って実装する
2. Phase 4で合意した設計方針に沿って実装する（設計フェーズを実行した場合）
3. **ライブラリ仕様の確認（必要な場合のみ）**: ライブラリのAPI仕様が不明な場合、ドキュメント参照MCP（context7 等）や公式ドキュメントで最新仕様を確認してから実装する
   - MCP未接続時は既存コードのパターンに倣い、不明点はユーザーに確認する
4. **中間報告（必須）**: 大きなファイル（概ね 300 行以上）や複数ファイルを一度に読み込んだ直後は、長時間の内部推論を始める前に以下を 2〜4 行でユーザーに報告する
   - 把握したこと（既存構造・関連箇所）
   - これから行う変更の計画
   - 迷いがある点があればその時点で質問する（推測で先に進まない）
5. コミットメッセージは `<type>: <subject> #<issue番号>` 形式で、Issue番号を必ず含める
   - 詳細は `.claude/rules/git-conventions.md`
   - **PreToolUse hook が `git commit` を検証する**ため、規約違反はブロックされる

## コミット粒度

- 論理的に独立した変更単位ごとにコミットする
- 1コミット = 1テーマを目安に、レビューしやすく分割する
- WIPコミットは避け、各コミットがビルド可能・テスト通過する状態を保つ

## グループ横断ユニットテストガイド

UI や手動での動作確認では「個別要素が期待通り動くか」は確認できるが、**類似要素のグループ全体に同じ仕様を適用したつもりが一部だけ反応しない** タイプのバグは検出が難しい。Phase 4 でグループ網羅性を要した設計の場合（`phases/04-design.md` 「必須チェック項目」参照）、ここで対応するユニットテストを必ず書く。

テスト記述パターン（Vitest / Jest の例）:

```ts
import { test, expect } from "vitest";

const TARGET_GROUP = ["item-a", "item-b", "item-c", "item-d"] as const;

test.each(TARGET_GROUP)("%s に新仕様が適用される", (id) => {
  const result = applyNewRule(id);
  expect(result.applied).toBe(true);
});
```

ポイント:

- **`test.each` でグループ要素の ID を回し、期待挙動を共通アサート** にする。新要素追加時にも配列に 1 行足すだけで自動的にテスト対象になる
- 既存テストファイルに同パターンが既にあれば踏襲する
- 手動・UI 検証は別途 Phase 6 で実施するが、グループ横断バグの **一次防御線はユニットテスト**

## スコープ外問題の記録と末尾提案

実装中に当該Issueのスコープ外の問題（周辺コードのバグ／仕様乖離／技術負債等）に遭遇した場合は **その場で直さず、メモとして記録するだけに留める**。1 Issue = 1 PR 原則を守り、実装フローを分断しないため。

1. **即時対応しない**: 当該Issueの完了条件（DoD）に含まれない変更は、ブランチ上で修正しない
2. **メモに記録する**: 発見時点で以下を控えておく
   - 何が問題か（1〜3行）
   - 該当ファイルと行番号（例: `src/data/items.ts:L481-498`）
   - バグの場合は再現手順
3. **Phase 5 末尾でまとめて提案**: 実装＋コミットが一通り完了したタイミングで、記録した問題を一括でユーザーに提示する
   - 起票判定基準は `SKILL.md` 「スコープ外問題の起票提案」節を参照
   - GitHub MCP の `search_issues` で既存Issueの重複を確認してから提案する
   - 候補ごとに Y/E/N の選択をユーザーに確認し、承認されたものだけ `issue_write`（method: `create`）で起票する
   - 本文末尾に `親Issue: #<親番号>` を明記する
4. **発見が無かった場合**: 「スコープ外問題の検出: なし」と明示的に報告する

> 例外: 実装の阻害になる問題（進行不能な致命バグ）に遭遇した場合は、即座にユーザーに報告し、当該Issueのスコープを広げるか・別Issueに切り出して先に対応するかを判断する。

## ワークフロー改善余地のメモ（Phase 8 向け）

実装中に「この手順は冗長だった」「このガードレールがあれば楽だった」「MCP ツールのこの挙動で詰まった」等、**ワークフロー全体（`/issue-start` 手順・ `.claude/rules/*`・`CLAUDE.md`・hooks・skills）への気づき** を感じた場合は、短文メモとして控えておく。

- **コード上のバグ**は前節の「スコープ外問題」として扱う（別Issue起票候補）
- **ワークフロー自体への気づき**は Phase 8 で知見ボードIssueにユーザー確認のうえ追記する（詳細は `.claude/rules/workflow-feedback.md` と `phases/08-issue-recording.md`）
- その場で直さない。メモだけ残してPhase 8 に引き継ぐ
