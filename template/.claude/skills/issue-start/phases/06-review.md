# Phase 6: コードレビュー（code-reviewer）

**目的**: 実装完了後、PRを作成する前にコード品質を確認する。バグや規約違反を早期に検出し、PRの品質を上げる。

## 実行手順

1. 実装が完了したら、**`code-reviewer` サブエージェントを3個並列で起動**する
   - `Agent` ツールで `subagent_type: "code-reviewer"` を指定する
   - エージェント定義は `.claude/agents/code-reviewer.md` に集約済み
2. 各エージェントには異なる観点でレビューさせる：
   - **シンプルさ・DRY・可読性**: 「コードの簡潔さ、重複排除、可読性の観点でレビューして」
   - **バグ・機能的正確性**: 「ロジックエラー、エッジケース、バグの観点でレビューして」
   - **プロジェクト規約・アーキテクチャ**: 「CLAUDE.md と .claude/rules/*.md の規約遵守、および既存アーキテクチャとの整合性をレビューして」
3. 信頼度80以上の指摘のみ集約し、重要度順にユーザーに提示する
4. **UI動作確認（フロントエンド変更を含む場合のみ）**: Playwright MCP 等で `browser_navigate` → `browser_snapshot` によりUI動作を確認する
   - 開発サーバーが起動済みの場合のみ実行する（起動していない場合はスキップ）
   - MCP未接続時はスキップし、ユーザーに手動確認を依頼する。**未検証である旨を完了報告に明記する**
5. **スコープ内／外の仕分け（必須）**: レビュアーの指摘を当該Issue DoD に照らして仕分ける
   - **スコープ内**: 当該PRで修正する（Phase 5 に戻って対応）
   - **スコープ外**: 起票候補として別管理する（1 Issue = 1 PR 原則を維持するため、当該PRに混ぜない）
   - 起票判定基準は `SKILL.md` 「スコープ外問題の起票提案」節を参照
6. **ユーザーの判断を仰ぐ**
   - スコープ内の指摘: 今すぐ修正 / 後で対応 / このまま進める
   - スコープ外の指摘: Y（そのまま起票）/ E（編集して起票）/ N（起票しない）
7. ユーザーの指示に従ってスコープ内の修正を適用する
8. 承認されたスコープ外の指摘は、GitHub MCP の `search_issues` で重複チェック後、`issue_write`（method: `create`）で子Issueとして起票する（本文末尾に `親Issue: #<親番号>` を明記）

## スキップ条件

- ユーザーが明示的にスキップを指示した場合
- ドキュメントのみの変更（`docs` ラベル）

## ワークフロー改善余地のメモ（Phase 8 向け）

レビュー観点の指摘とは別に、**レビュー手順自体やサブエージェント運用への気づき**（例: エージェント選択の基準が曖昧だった／並列数の妥当性／レビュー観点の追加候補）があれば短文メモで控え、Phase 8 で知見ボードIssueにユーザー確認のうえ追記する（詳細は `phases/08-issue-recording.md`）。
