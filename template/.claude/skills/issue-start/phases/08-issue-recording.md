# Phase 8: Issueへの記録

PR作成後、実装を通じて得られた情報をIssueにコメントとして記録する。後から同じIssueや関連作業を参照したときに、実装の経緯や判断が追えるようにすることが目的。

## 記録する内容

### 1. 実装メモ

実装中に判明した技術情報や設計判断。GitHub MCPの `add_issue_comment` で以下のフォーマットでコメントする：

```
## 実装メモ
- 採用した技術・ライブラリとその理由
- 設計上の判断とその根拠
- ディレクトリ構成やファイル配置の方針
```

### 2. ハマりどころ

トラブルシューティング情報。後から同じ問題に遭遇した際の参考になるよう、**症状・原因・解決策**の3点セットで書く。`add_issue_comment` で：

```
## ハマりどころ
**症状**: エラーメッセージや問題の現象
**原因**: なぜ起きたか
**解決**: どう直したか
```

### 3. その他の補足

利用規約上の注意点、外部サービスの制約など、コードだけでは読み取れない情報。

### 4. 起票した子Issueの記録（Phase 3/5/6 で子Issueを作成した場合）

Phase 3（探索）／Phase 5（実装）／Phase 6（レビュー）でスコープ外問題を子Issueとして起票した場合、親Issueに以下のフォーマットでコメント追記する。後から親Issueを見たときに、このIssue作業中に派生した関連作業が辿れるようにすることが目的。

```
## スコープ外で起票した子Issue

本Issueの作業中に発見したスコープ外の問題を、別Issueとして起票しました：

- #<子Issue番号> <タイトル> (<ラベル>) — 発見Phase: <3/5/6>
- #<子Issue番号> <タイトル> (<ラベル>) — 発見Phase: <3/5/6>
```

- GitHub MCP の `add_issue_comment` で親Issueにコメントする
- 起票した子Issueが複数ある場合は1コメントにまとめる
- 起票が発生しなかった場合はこのセクションを省略してよい

## 記録しなくてよいもの

- コードやコミット履歴から明らかに読み取れる内容
- PRの変更点に既に記載済みの内容

## 5. ワークフロー改善余地の知見ボードへの記録

個別Issue への記録（上記 1〜4）が終わった **最後** に、今回のセッション全体を振り返り、**ワークフロー自体（`/issue-start` の手順、`.claude/rules/*`、`CLAUDE.md`、hooks、skills、MCP 運用等）への改善余地があれば、知見ボードIssue（番号は `.claude/rules/workflow-feedback.md` 参照）にコメント追記する**。

目的は、次以降のセッションで同じ非効率を繰り返さないよう、気づきを永続化すること。運用規約の詳細は `.claude/rules/workflow-feedback.md` を参照。

### 記入対象

| 書く | 書かない |
|------|---------|
| ワークフローの冗長さ・曖昧さ | 個別Issueの実装メモ（→ 上記 1〜3） |
| スキル定義（SKILL.md / phases）の改善余地 | アプリ仕様そのもの |
| 規約・ガードレールへの追加要望 | コード上の個別バグ（→ 別Issue起票） |
| MCP 運用のハマりどころ・フォールバック提案 | コードスタイルの好み |

### 書き込み手順

1. **気づきの棚卸し**: Phase 5/6/7 の末尾で控えておいたメモを再確認する
2. **気づきが無い場合**: このステップ全体をスキップしてよい（「特になし」と明示コメントは不要）
3. **気づきがある場合**: 以下のフォーマットでユーザーに提示し、確認を得る

   ```
   【ワークフロー改善余地 候補】
   発生Phase: Phase N
   気づき: <1〜3行>
   現状の動作: <今のワークフローでどう進んだか>
   改善案: <どう変えるとよいか。SKILL.md / phases / rules / hooks のどこを変える想定か>
   重要度: Low / Medium / High（判定基準は workflow-feedback.md 参照）

   知見ボード #N に追記しますか？
    [Y] そのまま追記
    [E] 内容を編集して追記
    [N] 追記しない（口頭報告のみ）
   ```

4. **承認（Y/E）後の追記**: GitHub MCP の `add_issue_comment` で知見ボードIssueにコメントする。本文テンプレートは以下：

   ```markdown
   ## ワークフロー改善余地 [#<作業Issue番号>]
   **セッション**: https://claude.ai/code/session_xxxxx（可能であれば）
   **発生Phase**: Phase N（例: Phase 5 実装中）

   **気づき**: <何が非効率だった／躓いた／改善の余地があったか>

   **現状の動作**: <現在のワークフローではどう進んだか>

   **改善案**: <どう変えると良いか。SKILL.md / phases / rules / hooks のどこを変える想定か>

   **重要度**: Low / Medium / High
   ```

5. **複数ある場合**: 候補ごとにユーザー確認し、承認されたものだけを**それぞれ独立したコメント**として投稿する（1気づき = 1コメントを目安にする）

### 自動化しない

- **必ずユーザー確認を挟む**。無人書き込みはしない
- 知見ボードIssueは自動クローズしない（常時Open運用）

### 昇格後の扱い

ユーザーが知見ボードのコメントを独立した改善Issueに昇格させた場合、元コメントの末尾に `✅ Issue #XXX で対応` が追記される運用になっている（`.claude/rules/workflow-feedback.md`）。Claude Code 側で特別な扱いは不要。
