# Phase 3: コード探索（code-explorer）

**目的**: 実装に入る前に、既存コードの構造・パターン・関連箇所を深く理解する。ここでの理解が設計と実装の質を左右する。

## 実行手順

1. Issueの要件に基づき、**`code-explorer` サブエージェントを2〜3個並列で起動**する
   - `Agent` ツールで `subagent_type: "code-explorer"` を指定する
   - エージェント定義は `.claude/agents/code-explorer.md` に集約済み（システムプロンプトの再供給は不要）
2. 各エージェントには異なる観点で探索させる：
   - **類似機能の調査**: 「Issue要件に類似する既存機能の実装を包括的にトレースして」
   - **アーキテクチャ把握**: 「変更対象領域のアーキテクチャと抽象化を包括的にマッピングして」
   - **影響範囲の特定**: 「この変更が影響するコンポーネント・テスト・依存関係を特定して」
3. 各エージェントに「重要なファイル5〜10個のリストを含めて」と依頼する
4. **返却量の上限をプロンプトに明記する**（Stream idle timeout 回避。詳細は `@.claude/rules/context-efficiency.md`）：
   - 例: 「合計 400 行以内。各箇所は `file:line` ＋ 5 行以内の要約で。コード引用は 1 箇所 10 行以内」
   - 例: 「関数シグネチャと挙動要約（各 3 行以内）のみ。import 文や型定義の列挙は不要」
5. エージェントの結果を受け取ったら、**リストアップされたファイルを自分で読んで**理解を深める
   - ただし 500 行超のファイルは全読み禁止。Grep で該当行を特定してから `Read` に `offset`/`limit` を付けて必要範囲のみ読む
   - **中間報告（必須）**: 大量のファイル／大きなファイルを読み込んだ直後は、長時間の内部推論に入らず、読み取った内容の要点と「次に何をするか」を 2〜4 行でユーザーに報告してから次のステップに進む
6. **ライブラリ仕様の確認（必要な場合のみ）**: 実装で使用するライブラリに不明点がある場合、ドキュメント参照MCP（context7 等）や公式ドキュメントで最新仕様を確認する
   - MCP未接続時はスキップし、既存コードのパターンや公式ドキュメントURLをユーザーに共有する
7. 発見した内容をユーザーに簡潔に共有する（Phase 4 に進む前の **中間報告** を兼ねる）
   - 「把握したこと／未確定のこと／次の計画」の 3 点を明示する
   - 方針に迷いがある場合は推測で先に進まず、この時点でユーザーに質問する
8. **スコープ外問題の一覧化（必須）**: 探索中に当該Issueのスコープ外の問題（別バグ／設計上の違和感／データ誤り等）を見つけた場合、Phase末尾で以下を実施する
   - 起票判定基準（`SKILL.md` 「スコープ外問題の起票提案」節）に従って候補を絞り込む
   - GitHub MCP の `search_issues` で既存Issueの重複を確認する
   - 候補ごとに SKILL.md の起票フォーマットで提示し、Y/E/N の選択をユーザーに確認する
   - 承認された候補は `issue_write`（method: `create`）で子Issueとして起票する（本文末尾に `親Issue: #<親番号>` を明記）
   - 見つからなかった場合は「スコープ外問題の検出: なし」と明示的に報告する

## 並列起動の例

```
Agent(subagent_type: "code-explorer", prompt: "src/server/ 配下の<対象機能>系サービスをトレースして。エントリポイント・データフロー・依存関係を、各箇所 file:line + 3〜5 行の要約でまとめて。関連ファイル5〜10個のリスト付き。**合計 400 行・コード引用1箇所10行以内** を厳守。")
Agent(subagent_type: "code-explorer", prompt: "src/client/ の<対象UI>アーキテクチャを把握して。状態管理・APIコール・データ変換の要点を、各箇所 file:line + 3〜5 行で。関連ファイル5〜10個のリスト付き。**合計 400 行・import 文の列挙は不要** を厳守。")
```

## スキップ条件

以下の場合はこのPhaseをスキップして Phase 5（実装）に直行してよい：
- Issueラベルが `bug` で、修正箇所が明確な1ファイル程度の変更
- Issueラベルが `docs` のドキュメント変更
- ユーザーが明示的にスキップを指示した場合
