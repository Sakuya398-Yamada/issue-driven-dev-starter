# Phase 1: Issue分析 & 不足確認

Issueの内容を十分に把握してから実装に入ることが重要。曖昧なまま進めると手戻りが大きくなるため、ここで確実に情報を揃える。

## 0. 作業環境チェック（Phase 1 着手前）

セッション開始時の SessionStart hook 出力を必ず確認する。`## ⚠ Ghost worktree detected` が含まれていたら、**亡霊 worktree モード** で作業する（後述）。出力されていない場合でも、自分で確認したいときは以下のコマンドで判定できる。

```bash
pwd
git rev-parse --show-toplevel
git worktree list
```

判定マトリクス：

| pwd の状態 | 扱い |
|---|---|
| `git rev-parse --show-toplevel` と一致 | **メインリポ作業**: 通常通り |
| `git worktree list` の登録 worktree と一致 | **真の worktree 作業**: **Edit/Write の `file_path` は必ずこの worktree のパスを使う。メインリポ側のパスで指定すると pwd 基準の bash 実行と乖離して『編集したはずなのに反映されない』無音失敗を起こす** |
| `.claude/worktrees/<name>/` 配下なのに **登録 worktree でない** | **亡霊 worktree モード**: 以降のすべての Edit/Write/bash 相対パスをメインリポ絶対パス基準に切り替える |

### 亡霊 worktree モードでの作業ガイド

亡霊 worktree (`.claude/worktrees/<name>/` ディレクトリは残っているが `git worktree` には未登録の状態) は `.gitignore` 配下のため、その配下に書き込んだ Edit/Write は `git status` に出ず **無音で消える**。さらに bash 相対パスは pwd（亡霊側）基準で解決されるので、メインリポ側に書いた tmp ファイル等が辿れない事故が起きる。

このため：

- **`Edit` / `Write` の `file_path` は常にメインリポ絶対パスで指定する**（`worktrees/<name>/` をパスに含めない）
- **`Bash` での相対パスを避け、絶対パスを使う**
- **`git` 操作は `git -C "<main repo absolute path>" <subcommand>` 形式を常用する**（pwd に依存した暗黙の解決を避ける）

SessionStart hook が亡霊を検出したら警告セクションを出力する。警告がある時は迷わず上記モードに切り替えること（推測でメインリポ側か worktree 側か悩む必要はない）。

## 手順

1. GitHub MCPの `issue_read`（method: `get`）でIssueの本文・ラベルを取得し、`issue_read`（method: `get_comments`）でコメントも取得する
2. **マージ済みPRの確認**: GitHub MCPの `search_pull_requests` で関連PRの状態を確認する（query: `<issue番号>`、owner/repoを指定）
   - マージ済みPRが存在する場合、ユーザーに通知して追加作業の要否を確認する。不要であれば処理を終了する
   - Issue stateが `OPEN` でもPRがマージ済みのケースがあるので、Issue stateだけで判断しないこと
3. 関連する過去のIssue（本文中のリンクや関連ラベル）があれば参照する
4. **外部仕様・技術情報の検索（必要な場合のみ）**: Issueが外部サービスの仕様・ドメイン固有データ・ライブラリ仕様に関わる場合、Web検索MCP（または組み込みの `WebSearch`）で情報を収集する
   - **未接続・未設定時のフォールバック**: 組み込みの `WebSearch` → それでも得られなければユーザーに直接確認する
   - フロー全体を止めず、情報源を明示した上で次フェーズに進む
5. 以下のチェックリストでIssueの記載内容を検証する

## 必須チェックリスト

| 項目 | 確認内容 |
|------|---------|
| 背景・目的 | なぜこの作業が必要か明記されているか |
| 要件 | やること／やらないことが明確か |
| 完了条件 | 何をもって完了とするか定義されているか |
| 技術的な情報 | 実装に必要な設計・定義が十分か |

## Issue 本文と後追いコメントでスコープが乖離している場合の判定指針

Issue 本文 (作成時点の要件) と、後日追加されたコメント (実害バグの追加報告／追加要望／仕様変更) で扱う対象がずれている場合、以下のフローで判定する。

| コメントの性質 | 扱い |
|--------------|------|
| **本文の補足／不足分の補完** （仕様確認・抜けていた条件・参照データ等） | 本文スコープに **統合して** 進める |
| **本文と直交する別問題の報告** （別バグ／類似機能の別問題／追加要望） | **別Issue化を提案** し、本文スコープのみで進める |
| **本文の前提を覆す変更** （仕様変更・要件再定義） | ユーザーに「Issue 本文側を更新して進めるか／別Issueに切るか」を確認 |

判断は迷ったら **ユーザー確認を挟む**。無理に統合して PR が肥大すると 1 Issue = 1 PR 原則 (`.claude/rules/git-conventions.md`) が崩れる。

## 不足時の対応

チェックリストに不足がある場合、**実装に着手せず**ユーザーに聞き返す。

聞き返しのポイント：
- 何が不足しているか具体的に伝える
- 可能であれば選択肢や提案を添える
- ユーザーの回答を得てから次のPhaseに進む

**Example:**
```
Issue #3 の内容を確認しました。以下の点が不足しています：

1. **完了条件が未定義です**
   - 例: 「計算結果が±0.1%以内の誤差であること」のような基準はありますか？

2. **重複データの扱いが不明です**
   - 同一キーのレコードは上書き？スキップ？エラー？
```

## Phase 1.5: Issue自動補完（関連Issue & 完了条件）

実装に入る前に、Issueの「関連Issue」と「完了条件」を自動で補完する。ユーザーがこれらを毎回手書きする負担を減らすことが目的。

### 関連Issueの自動紐づけ

1. GitHub MCPの `list_issues` で全Issueを取得する（owner/repoを指定、perPage: 100）
2. 対象Issueの内容（タイトル・本文・要件）と各Issueを照合し、関連性を判定する
3. 判定基準：
   - 同じ機能領域に関わるIssue
   - 依存関係があるIssue（例: 前提となる機能、後続の拡張）
   - 同じコンポーネントを変更するIssue
4. 関連Issueが見つかった場合、GitHub MCPの `issue_write`（method: `update`）でIssue本文の「## 関連Issue」セクションを更新する

### 完了条件の自動生成

Issueの「やること」セクションの内容が「完了条件」セクションにそのままコピーされているか、完了条件が空・未定義の場合に自動生成する。

1. 「やること」の各項目を基に、検証可能な完了条件を生成する
2. 「やること」をそのまま完了条件にするのではなく、実装・動作確認の観点から条件を具体化する
3. 生成した完了条件でIssue本文の「## 完了条件（Definition of Done）」セクションを更新する

### 補完結果のユーザー確認

自動補完した内容をユーザーに提示し、問題がないか確認してから次のフェーズに進む。

**Example:**
```
Issue #10 の内容を自動補完しました：

**関連Issue:**
- #11 通知機能の追加（同じ設定画面に関連）
- #24 ユーザー設定の永続化（前提機能）

**完了条件（自動生成）:**
- [ ] 〇〇機能が正常に動作すること
- [ ] 既存のテストが通ること

この内容でIssueを更新してよいですか？
```

### Issue 本文 vs 現状コード乖離の検出

Issue 本文に「○○の追加が必要」「△△機構を新規実装する」のような **技術前提** が書かれている場合、実装着手前に該当機構の実在を `Grep` で確認する。本文が古いまま放置されていると、想定スコープと現状実装が大きく乖離する事態が起き得る。

確認手順:

1. 本文中の技術前提キーワード（型名・関数名・データキー・設定値等）を 2〜3 個抽出
2. `Grep -n "<keyword>"` で実在チェック。`type: "ts"` 等で対象を絞ると速い
3. **既に存在する**場合: スコープが「データ追加のみ」等に縮小し得るので、ユーザーに「Issueスコープを再確認したい」と提示する
4. **未実装が確認できた**場合: 通常通り Phase 3 / 4 に進む

省略する／しないの判断: Issue 本文の「やること」が **データ追加のみ** で技術前提を含まない場合はこのチェックを省いてよい。
